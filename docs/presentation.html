01 · The problem

Predict tomorrow's hourly load for the DE_LU grid zone

Every day before the deadline, submit 24 hourly values — tomorrow 00:00 to 23:00 UTC, in megawatts — for the Germany/Luxembourg bidding zone.

  • Scoring: a live leaderboard scores every team's daily CSV against the actuals once they materialise, and ranks by mean MAE. You can't overfit a hidden test set — the test set is tomorrow.
  • Two reference marks: ENTSO-E's own published day-ahead forecast is scored alongside the teams, and a 7-day seasonal naive is the internal floor every model must beat.

Input signals

ENTSO-E actual load · ENTSO-E's own published day-ahead forecast (DAF) · Open-Meteo weather (15 variables) · calendar structure (hour, weekday, month, holidays)

Output

24 strictly positive MW values, one per hour, rounded to 2 decimals, delivered as a pull request to the leaderboard repository.

02 · The data

Three sources, one canonical hourly-UTC grid

ENTSO-E — actual load load_mw

The target. 15-minute Europe/Berlin readings, Jan 2022 → Apr 2026, resampled to hourly UTC. 37,944 clean rows.

ENTSO-E — day-ahead forecast forecast_mw

The operator's own published forecast. Never a model input — it is the benchmark to beat.

Open-Meteo — weather 15 columns

Temperature, humidity, precipitation, cloud cover, wind, pressure… for a Frankfurt reference point. Archive for history, forecast endpoint for the future hours the model must predict into. No API key.

  • Cleaning is conservative and loud: IQR outlier bounds (multiplier 3.0) — chosen over the library's IsolationForest because IQR is deterministic (CR-2).
  • Gaps ≤ 3 h are linearly interpolated; anything longer raises — the surrounding window is rejected rather than fabricated (CR-3).
  • "Load is the spine": cleaned load defines the timestamp grid. A gap in load is fatal; DAF and weather gaps are soft-dropped with a warning.
  • Asymmetric trim: DAF is reindexed onto the load grid (it exists to be compared with actuals) — but weather keeps its forecast tail beyond the last actual, because predicting tomorrow needs tomorrow's weather.

03 · Architecture

Seven stages that only talk through Parquet

Each stage is independently runnable, testable, and auditable. No stage calls another in memory — state flows through files, via one storage abstraction.

→

→

one forecast block per fold
→

→

Plotly / Matplotlib backends

format-submission

Seam 1 — DataStore

Two methods: read(name) / write(name, df). All pipeline I/O funnels through it — storage is swappable and every access is one auditable chokepoint.

Seam 2 — DataSource

Where download gets raw frames: the live APIs in production, or seeded offline fixtures in CI. The stage can't tell the difference — so the whole pipeline runs end-to-end in CI with no network and no key.

04 · The model

Gradient-boosted trees, applied recursively

  • Estimator: LGBMRegressor inside ForecasterRecursive from spotforecast2-safe — predicts hour t+1, feeds it back as input for t+2, … out to the horizon.
  • Autoregression: lags=168 — the previous full week of hourly load as features. Load is dominated by daily and weekly rhythm.
  • Determinism flags (CR-2): random_state=42, deterministic=True, force_col_wise=True — retraining on identical data reproduces the identical model.
  • Deliberately not done (yet): hyperparameter tuning, prediction intervals, SHAP attribution — documented as open limitations in the model card, not quietly skipped.

40 exogenous features per hour

Calendar (25): smooth RBF encodings of hour (10), weekday (7), month (6) — so 23:00 sits mathematically next to 00:00 — plus German holiday and weekend flags.

Weather (15): the Open-Meteo columns, joined on the hourly index. Any NaN in the joined matrix raises (CR-3).

Why not the library's all-in-one model class?

ForecasterRecursiveModel does its own data loading — it would bypass the DataStore audit seam. The project uses the lower-level fit/predict pieces instead.

05 · Evaluation protocol

365 rolling origins

A single train/test split can flatter or damn a forecaster by chance. The protocol here re-runs the whole exercise at 365 evenly spaced daily origins across the final year of data.

  • One fold = one origin. Train on everything before the origin (expanding window), forecast the next 24 h as a half-open block [start, end), score it against the actuals.
  • One model per fold, persisted under models/folds/<fold_id>/ — no leakage between folds is possible by construction.
  • Fold generation is fail-safe: unsorted index, origins off the hourly grid, a span that doesn't divide evenly, or less than 168 h of history — every case raises with a specific message. Strides are computed in integer nanoseconds (CR-2).
  • Production reuses the machinery: the daily job pins n_forecasts=1 at the freshest actual — the operational forecast is just a one-fold backtest whose block covers tomorrow.

Scored against two benchmarks

7-day seasonal naive — tomorrow 3 pm = last Tuesday 3 pm. The honesty floor: a model that can't beat it has learned nothing.

ENTSO-E DAF — the official operator forecast. The professional ceiling.

Four metrics, one primary

MAE (primary — robust, in interpretable MW) plus RMSE, MAPE, SMAPE, each aggregated to mean ± std ± median across folds, with skill scores 1 − MAE_model / MAE_benchmark.

07 · Production

A forecast ships fully automated every day at 17:00 UTC

GitHub Actions runs the real pipeline against the live APIs and opens a pull request on the leaderboard.

  • Fetch fresh: DOWNLOAD_END = today; then a freshness gate asserts the newest actual is less than an hour old — stale data aborts the run.
  • Pin the origin: the fold set collapses to exactly one fold anchored at the freshest actual, with a 48 h horizon so the block safely covers all of tomorrow.
  • Train → predict → validate: one model, one forecast; the submission slice must be exactly 24 rows, NaN-free, finite, strictly positive — or it raises (CR-3).
  • Submit as a PR from the team's fork to the upstream leaderboard repo.

40 daily forecasts shipped — every scheduled run green

A scheduled run trains, predicts, validates the 24-row slice, and opens the leaderboard PR — unattended, in 2–4 minutes.

08 · The constraint that shaped everything

Treated as a high-risk AI system under the EU AI Act

Load forecasting for critical infrastructure falls under Annex III No. 2. Articles 9–15 apply — and were operationalised as four non-negotiable code rules enforced on every module.

Rule Name What it means in the code AI Act
CR-1 No dead code Every function and branch is reached by a test or docstring example. Art. 9, 13
CR-2 Determinism Same input → bit-identical output. Fixed seeds, fixed iteration order, pinned data windows, integer-nanosecond fold strides. Art. 12, 15
CR-3 Fail-safe processing Invalid input (NaN, wrong dtype, gaps) raises a specific exception — never silently imputed. Art. 10, 15
CR-4 Minimal attack surface One vetted library (spotforecast2-safe) on the production path; SHA-pinned CI actions; least-privilege tokens. Art. 15

Plus: all storage behind a DataStore protocol · every run leaves audit artifacts · rolling-origin backtesting instead of a single split (Art. 9) · model cards for transparency (Art. 13).

09 · Two production incidents

Real example

The vanished weather tail PR #28

Daily run #12 died: "Exog matrix has 720 NaN values" — 48 hours × 15 weather columns. A merge refactor had trimmed weather to the load's time range, deleting the future weather the forecast needs (in production the entire forecast window lies past the last actual). The e2e test missed it because its window sat inside the load range.

Fix: trim weather only at the start, keep the forecast tail. Lesson: DAF and weather need opposite alignment rules — now an explicit, tested asymmetry.

The wind outage PRs #44 · #45

In July 2026, Open-Meteo's archive stopped serving 10 m wind. Because missing data raises instead of imputing (CR-3), the pipeline failed loudly rather than silently training on NaNs.

Fix: a two-line, dated change dropping the wind columns — reverted two lines when the archive recovered eight days later. Lesson: fail-safe + one source of truth for the column list turns an upstream outage into a trivially reviewable patch.

11 · What's next

The known levers, in order

  • Zone-wide weather — one Frankfurt station currently represents all of DE_LU; multi-station aggregation is the most probable accuracy lever vs the leaders.
  • Hyperparameter tuning — LightGBM runs library defaults plus determinism flags; nothing has been tuned yet.
  • Holiday & Monday error tail — the worst live days are public holidays (Pentecost Monday 6,215 MW, Corpus Christi 4,227 MW) and Mondays; richer holiday/bridge-day features would cut the tail of the daily-MAE distribution.
  • Prediction intervals — point forecasts today; operators want uncertainty.

What this project demonstrates

A regulated-grade forecasting system, not a notebook: deterministic end-to-end, fail-safe at every boundary, evaluated by a 364-fold rolling-origin protocol — beating the official ENTSO-E forecast both in backtest and on the live leaderboard — and submitting a real forecast every single day.
